10   Advantages and Limitations

10.1   Advantages

Minimal hardware requirements. The complete system operates on a commodity laptop with a $15 USB wireless adapter, making it accessible to students and under-resourced laboratories without dedicated network appliance budgets.

Closed-loop transparency. Every detection, scoring, decision, and enforcement action is recorded with full evidential context. A reviewer can inspect not only what the system did but precisely why it acted, including rule scores, threshold comparisons, and decision rationale strings.

Graduated response with reversibility. The five-tier response policy (allow, log, rate-limit, redirect, block) reduces the impact of misclassification compared to binary allow/block systems. Blocked devices retain Wi-Fi association and dashboard visibility, and containment can be reversed through a single administrative action.

Integrated evidence pipeline. Honeypot sessions, packet metadata, IOC extraction, and threat events are stored in a unified database and accessible through a single dashboard, eliminating the log correlation overhead typical of multi-tool deployments.

Research instrumentation. Built-in telemetry recording enables quantitative evaluation of detection accuracy, false-positive rates, latency distributions, and resource consumption without requiring external measurement infrastructure.

10.2   Limitations

Limited detection scope. Six heuristic detectors cover common reconnaissance and volumetric attacks but do not address application-layer exploits, protocol-specific evasion techniques, encrypted payload inspection, or sophisticated slow-rate attacks that operate below sliding-window thresholds.

Single point of failure. The gateway concentrates all defensive functions on a single host. Hardware failure, kernel panic, or process crash disables all detection, containment, and evidence collection simultaneously. Production deployments would require redundancy mechanisms absent from this prototype.

IP-based containment granularity. Containment rules target source IP addresses. In environments with DHCP address rotation, containment persistence may degrade if a blocked device obtains a new address. MAC-aware containment is not implemented in the current version.

IPv6 and tunnel limitations. The prototype tracks IPv6 packets and flags VPN-associated UDP ports for visibility but does not perform deep IPv6 inspection, fragment reassembly, or tunnel content analysis. Attackers using IPv6-only traffic or encrypted tunnels can evade detection.

Honeypot fingerprinting. Cowrie and the lightweight TCP decoys may be identifiable by sophisticated attackers through timing analysis, banner inconsistencies, or interaction depth limitations. The honeypot layer is effective against automated tools and commodity scanners but should not be relied upon for advanced adversary engagement.

Evaluation scale. Experiments were conducted with two mobile devices on an isolated subnet. While this validates the architecture's functional correctness, it does not demonstrate performance under production-scale traffic loads with dozens or hundreds of concurrent clients. Scalability evaluation remains future work.

NAT and firewall consolidation. The current deployment uses iptables for NAT masquerading alongside nftables for dynamic security rules. This dual-framework configuration introduces operational complexity. Full migration to nftables-only configuration is a planned improvement.

11   Conclusion

This paper presented NTTH, an event-driven security gateway prototype that integrates packet observation, rule-assisted risk scoring, graduated forwarding-layer containment, and honeypot-based evidence collection into a single deployment operating on commodity laptop hardware. The architecture addresses a documented gap in the availability of reproducible, low-cost defensive loop implementations suitable for university-scale security education and applied research.

Controlled experiments conducted with two mobile devices on an isolated wireless network demonstrated that the prototype successfully detects port scans, host sweeps, SYN floods, and brute-force attempts with a 100% detection rate across fifteen attack trial executions. No false-positive block or rate-limit actions occurred during forty minutes of cumulative benign traffic workloads including idle state, web browsing, video streaming, and application updates. Average capture-to-enforcement latency remained below 900 milliseconds across all enforcement scenarios, with the system consuming less than 380 MB of resident memory.

The principal contribution is not a novel detection algorithm but rather a systems architecture that demonstrates how existing primitives — packet capture via Scapy, sliding-window heuristics, policy-driven response, nftables enforcement, Cowrie honeypot, and real-time web dashboard — can be composed into a measurable and transparent closed-loop defense. The complete system, including its threshold configuration, detection heuristics, decision policy, and enforcement mechanisms, is documented at a level sufficient for independent reproduction and extension.

NTTH occupies a deliberate position between production-grade security appliances and academic proof-of-concept demonstrations. It provides enough operational capability to support meaningful security experiments and educational exercises while maintaining architectural simplicity that enables understanding, modification, and extension by researchers and students.

12   Future Work

Several directions for future development emerge from the current limitations. First, MAC-address-aware device identity tracking would improve containment persistence across DHCP lease changes and enable device-level rather than IP-level policy enforcement. Second, complete migration from the current iptables/nftables hybrid to a unified nftables-only configuration would simplify operational management and reduce potential rule conflicts.

Third, integration of lightweight machine-learning anomaly detection models trained on the gateway's own baseline traffic would activate the anomaly score component of the risk calculation formula, potentially improving detection of novel or slow-rate attacks not covered by the current heuristic set. Public datasets such as CICIDS2017 [17] and Edge-IIoTset [8] could supplement training data.

Fourth, IPv6 deep inspection, IP fragment reassembly, and VPN/tunnel content analysis would address current visibility gaps. Fifth, the honeypot layer could be enhanced with protocol-specific interaction depth — more realistic MySQL, FTP, and Redis emulation — to improve evidence quality against moderately sophisticated adversaries.

Sixth, scalability evaluation with larger client populations (10–50 concurrent devices) and longer deployment durations (multi-day soak testing) would validate the architecture's applicability beyond the current two-device experimental scope. Finally, comparative benchmarking against integrated open-source security stacks such as Security Onion running identical attack scenarios would provide quantitative positioning data for the research community.
